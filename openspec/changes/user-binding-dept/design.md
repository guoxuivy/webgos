## Context

当前项目已实现部门管理的基础功能，包括部门的增删改查和树形结构展示。用户表已有 `department_id` 字段用于关联部门，但缺少完整的用户部门关系绑定管理能力。

## Goals / Non-Goals

**Goals:**
- 在部门列表页面展开时显示下级部门和直接成员列表
- 在部门编辑页面支持批量添加员工（从用户表选择）
- 在部门编辑页面支持批量移除员工
- 新增后端API支持批量用户操作

**Non-Goals:**
- 不支持一个用户属于多个部门（保持一对一关系）
- 不涉及权限控制逻辑变更
- 不修改数据库表结构（使用现有字段）

## Decisions

1. **用户部门关系模型**：采用一对一关系，用户表的 `department_id` 字段存储所属部门ID，部门删除时将用户的 `department_id` 置为0

2. **批量操作API设计**：
   - `POST /api/department/{id}/users`: 批量添加用户到部门
   - `DELETE /api/department/{id}/users`: 批量从部门移除用户

3. **前端展示方案**：
   - 使用树形表格展示部门结构
   - 展开部门节点时显示成员列表
   - 编辑页面使用多选组件选择用户

4. **数据一致性**：移除用户时检查用户确实属于该部门，避免误操作

## Risks / Trade-offs

- **性能风险**：大量用户操作时可能影响性能 → 使用批量SQL操作优化
- **数据一致性**：并发操作可能导致数据不一致 → 使用事务保证完整性
- **前端展示**：部门树包含成员后可能影响加载速度 → 按需加载成员数据
